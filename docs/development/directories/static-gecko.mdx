## Overview

The `static/gecko/` directory is the foundation of Floorp's customized Firefox build. It houses patched Gecko platform assets, preference overrides, and version configuration that are applied during the build and packaging pipelines. In Floorp's layered architecture, this directory constitutes the **Firefox Base** layer — the lowest layer upon which ESM modules, the bridge, Chrome UI, and pages are built (`static/gecko/pref/override.ini`).

## Architecture Role

Floorp is structured in five layers, and `static/gecko/` sits at the base:

1. **Firefox Base** — Patched Gecko and pref overrides, sourced from `static/gecko/`.
2. **ESM Modules** — Firefox ESM modules and Window Actor registration built on top of the base Gecko layer.
3. **Bridge** — Startup bridge that selects dev/test HTTP loaders or `chrome://` production code.
4. **Chrome UI** — SolidJS browser chrome features discovered by glob.
5. **Pages** — React/Tailwind full-page browser UIs.

The Firefox Base layer provides the patched Gecko engine and preference configuration that all upper layers depend on (`static/gecko/pref/override.ini`).

## Preference Overrides

The `pref/override.ini` file within `static/gecko/` defines preference overrides that are applied to the Firefox base configuration. This file is the canonical source for Floorp's default preference modifications — it ensures that Floorp-specific defaults (such as update channels, UI behavior, and feature toggles) are baked into the build rather than applied at runtime (`static/gecko/pref/override.ini`).

During the packaging workflow, preference overrides are applied to the Firefox preference file (`firefox.js`) as part of the build process. The packaging CI workflow applies these overrides before the final installer is assembled (`.github/workflows/package.yml`).

## Version Configuration

The `static/gecko/` directory contains version configuration that drives the build identity. During packaging, the CI workflow reads version and version-display files from `static/gecko/config/` and writes combined version strings into an autogenerated configuration directory. The build system uses `--with-version-file-path` pointing to the autogenerated config under `static/gecko/config/` to inject Floorp-specific version metadata into the Gecko build (`.github/workflows/package.yml`).

The version files under `static/gecko/config/` store the Floorp (Noraneko) version independently from the upstream Firefox version. During packaging, both values are combined — the Floorp version and the Firefox version are concatenated into the autogenerated version files that the Gecko build system consumes (`.github/workflows/package.yml`).

## Branding Resources

The packaging workflow contains branding-copy logic, but the current checkout does not treat a branding directory under `static/gecko` as a required source directory. The workflow first checks for repository-provided branding assets before copying them into Firefox's `browser/branding` tree, and it also contains a guarded compatibility check for an optional Noraneko-side branding directory (`.github/workflows/package.yml`). Because that optional directory is not present in this checkout, documentation should cite the workflow file rather than treating the optional branding location as a source-owned directory.

The packaging workflow also supports conditional branding selection. Depending on whether the build targets an alpha or release channel, the workflow selects different branding directories and applies the corresponding branding configuration to the build. This is controlled through mozconfig options that specify the branding directory path (`.github/workflows/package.yml`).

## Interaction with the Build Pipeline

The `static/gecko/` directory is consumed at multiple stages of the Floorp build pipeline:

### Pre-build Configuration

Before the Gecko build is configured, the packaging workflow prepares branding conditionally and writes version files to the autogenerated config path under `static/gecko/config/`. The mozconfig is then assembled with options that reference these generated version files and the selected browser branding directory (`.github/workflows/package.yml`).

### Preference Application

After the artifact build is unpacked, preference overrides from `static/gecko/` are applied to the Firefox default preferences file. This step ensures that the packaged binary ships with Floorp's preferred default settings rather than upstream Firefox defaults (`.github/workflows/package.yml`).

### Packaging and MAR Generation

The preference-overridden and branding-customized build output is then packaged into platform-specific installers and MAR update archives. The version metadata from `static/gecko/config/` is embedded into the MAR directory structure, including build IDs and version strings that are used for update manifests (`.github/workflows/package.yml`).

## Relationship to Upper Layers

While `static/gecko/` provides the base Gecko engine and its configuration, the upper layers of Floorp build upon it:

- **ESM modules** in `browser-features/modules/` register Window Actors and extend the Gecko browser glue logic, operating on top of the patched Gecko base (`browser-features/modules/modules/BrowserGlue.sys.mts`).
- **The bridge** in `bridge/startup/` selects between development HTTP loaders and production `chrome://` URLs, but the underlying browser process is the patched Gecko from `static/gecko/` (`bridge/startup/src/chrome_root.ts`).
- **Chrome features** and **pages** are loaded into the browser chrome and content areas, respectively, running within the Gecko environment established by this base layer (`browser-features/chrome/common/mod.ts`).
