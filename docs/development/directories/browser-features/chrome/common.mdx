## Overview

The `browser-features/chrome/common/` directory contains Floorp's runtime-oriented browser chrome features. Each feature lives in its own subdirectory with an `index.ts` entrypoint and is discovered automatically by a glob pattern. This page documents the discovery mechanism, the shared module contract, and the individual feature modules that ship under `common/`.

## Feature Discovery

Common chrome features are discovered by a glob scan over the `common/` directory. The discovery pattern is defined in `browser-features/chrome/common/mod.ts` as `./*/index.ts`, meaning every immediate child directory that contains an `index.ts` file is treated as a registrable feature module.

This glob-based approach means that adding a new feature only requires creating a new subdirectory with an `index.ts` that exports the expected entrypoints. There is no central hand-maintained registry list in `mod.ts`; the directory structure itself is the registry.

The loader layer then builds module maps from these chrome feature entry globs, as implemented in `bridge/loader-features/loader/modules.ts`. At runtime, the loader entrypoint in `bridge/loader-features/loader/index.ts` filters enabled modules and runs their lifecycle hooks.

## Module Contract

Every common chrome feature module follows a consistent entrypoint contract:

- A **default class export** that owns the feature's lifecycle. The class name typically matches the feature name (for example, `Workspaces`, `TabBar`, `StatusBar`).
- An **`init` method** that performs runtime initialization, such as wiring components, services, and styles.
- Some modules expose an additional **`initBeforeSessionStoreInit`** method for initialization that must occur before the session store is restored. For example, the `ui-custom` feature exposes both `init` and `initBeforeSessionStoreInit` (`browser-features/chrome/common/ui-custom/index.ts`).

Modules typically wire internal sub-resources such as services, components, configuration objects, and inline styles. These are referenced as relative imports within each feature directory (for example, `service`, `config`, `components/*`, `style.css?inline`).

## Relationship to the Loader

The common chrome features are consumed by the bridge loader infrastructure. The module map construction in `bridge/loader-features/loader/modules.ts` builds common and static module maps from chrome feature entry globs. The loader entrypoint in `bridge/loader-features/loader/index.ts` then imports `BrowserGlue`, initializes i18n, records loader state, filters enabled modules, and runs module lifecycle hooks.

Module load-state resolution and `onModuleLoaded` promises are provided by `bridge/loader-features/loader/modules-hooks.ts`, which the loader uses to coordinate dependencies between feature modules during initialization.

The production loader path is `chrome://noraneko/content/core.js`, while development and test environments use HTTP-based loaders served from port 5181, as configured in `bridge/startup/src/chrome_root.ts` and `bridge/loader-features/vite.config.ts`.

## Relationship to Static Chrome Features

In addition to the `common/` directory, Floorp maintains a `browser-features/chrome/static/` directory for features that require pre-session or static initialization. Static feature discovery is provided by `browser-features/chrome/static/mod.ts`. The static features include `downloadbar`, `overrides`, and `prefs`, where `prefs` also exposes an `initBeforeSessionStoreInit` entrypoint. Both common and static module maps are built together by the loader's module registry.
