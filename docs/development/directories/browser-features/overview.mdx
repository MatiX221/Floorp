# Browser Features Overview

The `browser-features` directory is the central location for Floorp's custom browser functionality built on top of the Firefox base. It is organized into three major subdirectories that correspond to distinct architectural layers: `chrome` for SolidJS-based browser chrome UI, `pages-settings` for React-based full-page settings UI, and `modules` for Firefox ESM system modules and Window Actor registration.

## Architectural Layers

Floorp's browser features sit between the patched Gecko base and the bridge loader that injects them into the browser at runtime. The layers within `browser-features` are:

1. **Chrome UI** — SolidJS browser chrome features discovered by glob from `browser-features/chrome/common/mod.ts`. These features augment the native browser chrome with custom UI components and services.
2. **Pages** — React/Tailwind full-page browser UIs, such as the settings application, configured via `browser-features/pages-settings/vite.config.ts`.
3. **ESM Modules** — Firefox ESM modules and Window Actor registration, defined in `browser-features/modules/modules/BrowserGlue.sys.mts`.

The bridge loader is responsible for discovering, filtering, and initializing these features at runtime. The loader entrypoint at `bridge/loader-features/loader/index.ts` imports BrowserGlue, initializes i18n, records loader state, filters enabled modules, and runs module lifecycle hooks. The module registry at `bridge/loader-features/loader/modules.ts` builds common and static module maps from chrome feature entry globs, while `bridge/loader-features/loader/modules-hooks.ts` provides `onModuleLoaded` promises and internal module load-state resolution.

## Chrome Features

The `browser-features/chrome` directory contains Floorp's custom browser chrome features. It is divided into `common` features that are dynamically loaded and `static` features that are initialized at fixed lifecycle points. The overview and utility conventions for these directories are documented in `browser-features/chrome/README.md`.

### Feature Discovery

Chrome features are discovered using a glob pattern. The common module registry at `browser-features/chrome/common/mod.ts` uses the glob pattern `./*/index.ts` to discover all common chrome feature entry points. Static chrome feature discovery is handled separately by `browser-features/chrome/static/mod.ts`.

The loader module registry at `bridge/loader-features/loader/modules.ts` consumes these globs to build the module maps that the loader entrypoint iterates over at runtime.

### Common Chrome Features

Common chrome features are SolidJS-based modules that own individual browser chrome capabilities. Each feature is implemented as a default-exported class with an `init` method, and some features expose additional lifecycle entrypoints such as `initBeforeSessionStoreInit`. ### Module Lifecycle

The loader entrypoint at `bridge/loader-features/loader/index.ts` orchestrates the module lifecycle. After importing BrowserGlue and initializing i18n, it records the loader state, filters the discovered modules to only those that are enabled, and then runs each module's lifecycle hooks. The `onModuleLoaded` promises provided by `bridge/loader-features/loader/modules-hooks.ts` allow the loader and other components to react to individual module load completion, with internal load-state resolution tracking the progress of each module.

Some features, such as `ui-custom` and `prefs`, expose an `initBeforeSessionStoreInit` entrypoint in addition to the standard `init` method. This allows those features to perform initialization work before the session store has finished restoring state, which is necessary for features that need to manipulate preferences or DOM layout early in the startup sequence.

## Pages: Settings Application

The `browser-features/pages-settings` directory contains a React and Tailwind CSS application that provides Floorp's full-page settings UI. This is a separate build target from the chrome features, with its own Vite configuration at `browser-features/pages-settings/vite.config.ts`.

### Application Entry

The settings application is bootstrapped by `browser-features/pages-settings/src/main.tsx`, which initializes the React app with theme support, i18n, and hash-based routing.

## ESM Modules and Window Actors

## Relationship with the Bridge Loader

The bridge startup layer at `bridge/startup/src/chrome_root.ts` selects the appropriate loader based on the runtime environment. In development, it uses an HTTP loader served from a Vite dev server; in production, it loads the bundled chrome code. The loader then discovers and initializes the browser features described in this page.

The flow from startup to feature initialization is:

1. The bridge startup at `bridge/startup/src/chrome_root.ts` selects the loader URL based on the environment.
2. The loader entrypoint at `bridge/loader-features/loader/index.ts` imports BrowserGlue from `browser-features/modules/modules/BrowserGlue.sys.mts`, which registers the Window Actors.
3. The loader module registry at `bridge/loader-features/loader/modules.ts` builds module maps from the chrome feature entry globs discovered by `browser-features/chrome/common/mod.ts` and `browser-features/chrome/static/mod.ts`.
4. The loader filters enabled modules and runs their lifecycle hooks, with load-state tracked by `bridge/loader-features/loader/modules-hooks.ts`.
5. The settings page application at `browser-features/pages-settings/src/main.tsx` is loaded as a separate full-page UI when the user navigates to the settings page.
