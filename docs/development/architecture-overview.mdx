## Architecture Overview

Floorp is a Firefox-derived browser that layers custom UI, modules, and tooling on top of a patched Gecko engine. The codebase is organized into discrete layers that each own a distinct responsibility, from preference overrides at the engine boundary to React-based full-page settings UIs.

### Layered Model

The architecture can be understood as five stacked layers, each with a clear entrypoint and boundary:

1. **Firefox Base** â€” Patched Gecko engine and preference overrides that establish the runtime baseline (`static/gecko/pref/override.ini`).
2. **ESM Modules** â€” Firefox ES Module system modules and Window Actor registration that bridge Gecko internals to Floorp features (`browser-features/modules/modules/BrowserGlue.sys.mts`).
3. **Bridge** â€” Startup bridge that selects between dev/test HTTP loaders and production `chrome://` code (`bridge/startup/src/chrome_root.ts`).
4. **Chrome UI** â€” SolidJS-based browser chrome features discovered by glob from `browser-features/chrome/common/mod.ts`.
5. **Pages** â€” React/Tailwind full-page browser UIs such as the settings application (`browser-features/pages-settings/vite.config.ts`).

### Firefox Base and Preference Overrides

The lowest Floorp-owned layer sits at the Gecko preference boundary. The override file in `static/gecko/pref/override.ini` provides patched preference defaults that are applied before Floorp-specific modules execute. During packaging, the CI pipeline applies a preference override script to `firefox.js` in the distribution directory, confirming that this layer is integrated into the build rather than only present in source.

### Bridge and Loader

The bridge layer is the startup entrypoint that decides how Floorp's feature code is loaded into the browser chrome.

#### Loader Selection

`bridge/startup/src/chrome_root.ts` (line 59) defines three loader paths:

- **Dev loader**: `http://localhost:5181/loader/index.ts` â€” used during local development with hot reloading.
- **Test loader**: `http://localhost:5181/loader/test/index.ts` â€” used by the colocated test runner for browser integration tests.
- **Production loader**: `chrome://noraneko/content/core.js` â€” the bundled production entrypoint served from the browser's chrome protocol.

The dev/test loader is served by a Vite dev server configured on port 5181 (`bridge/loader-features/vite.config.ts`, line 17).

#### Loader Entrypoint

The loader entrypoint at `bridge/loader-features/loader/index.ts` performs the following sequence:

1. Imports `BrowserGlue` to ensure ESM module actors are registered.
2. Initializes internationalization.
3. Records loader state for diagnostics.
4. Filters enabled modules based on preferences and feature flags.
5. Runs module lifecycle hooks for each enabled module.

#### Module Registry

`bridge/loader-features/loader/modules.ts` builds two module maps â€” common and static â€” from chrome feature entry globs. These maps are consumed by the loader to determine which feature modules to instantiate.

#### Module Hooks

`bridge/loader-features/loader/modules-hooks.ts` provides `onModuleLoaded` promises and internal module load-state resolution. This allows features to depend on the readiness of other features without introducing hard coupling.

### Chrome UI Features

The Chrome UI layer contains SolidJS-based browser chrome features. These are divided into two categories: **common** features that are dynamically discovered and **static** features that are initialized at fixed lifecycle points.

#### Common Feature Discovery

`browser-features/chrome/common/mod.ts` (line 4) discovers common features using the glob pattern `./*/index.ts`. Each discovered feature exports a default class and an `init` function that the loader calls during the module lifecycle.

### Data Flow Summary

At startup, the following sequence occurs:

1. Gecko loads with Floorp preference overrides from `static/gecko/pref/override.ini`.
2. `BrowserGlue.sys.mts` registers 22 JS Window Actors via `ActorManagerParent.addJSWindowActors`.
3. The bridge in `bridge/startup/src/chrome_root.ts` selects the appropriate loader based on environment (dev HTTP, test HTTP, or production `chrome://`).
4. The loader at `bridge/loader-features/loader/index.ts` imports BrowserGlue, initializes i18n, filters enabled modules, and runs lifecycle hooks.
5. The module registry at `bridge/loader-features/loader/modules.ts` provides common and static module maps built from feature entry globs.
6. Common chrome features are discovered via `./*/index.ts` in `browser-features/chrome/common/mod.ts` and initialized through their `init` entrypoints.
7. Static chrome features are initialized at fixed lifecycle points, with `prefs` and `ui-custom` running pre-session-store hooks.
8. Full-page UIs such as settings are loaded on demand through the Pages layer, with React routing defined in `browser-features/pages-settings/src/App.tsx`.
